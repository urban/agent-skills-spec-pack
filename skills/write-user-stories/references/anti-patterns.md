# Anti-patterns

Rewrite or reject stories that:

- start with implementation details
- use the system as the actor by default without a real need
- omit the situation or trigger
- omit the reason the behavior matters
- omit an observable result
- combine several behaviors into one story
- describe tasks rather than outcomes
- sound like generic Agile boilerplate
- claim invisible success such as "the system processes correctly" without an external sign
- hide safety, trust, review, or control boundaries that should be visible in the stories
