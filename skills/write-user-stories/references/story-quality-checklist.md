# Story quality checklist

A strong story should:

- identify a clear actor
- describe a concrete situation or trigger
- describe one meaningful action
- explain the value of the behavior through the outcome
- include an observable result
- avoid implementation detail unless directly user-visible
- be small enough to build and test
- avoid heavy overlap with another story
- map cleanly to acceptance criteria or implementation slices
- use `TODO: Confirm` instead of guessed intent when evidence is weak
